Module 1 — Why RAG exists

The problem plain LLMs cannot solve on their own

Analogy: an LLM without RAG is like a brilliant consultant who hasn't read your company's files. They're smart, but they can only draw on what they studied years ago — not on last quarter's policy update sitting in your Confluence.

Core insight: LLMs predict probable text; they do not retrieve verified facts. RAG separates retrieval (finding the right passage) from generation (writing a fluent answer) so each can be evaluated and improved independently.

Module 2 — The offline pipeline

How raw documents become searchable chunks

This pipeline runs on a schedule (e.g. weekly). It is completely separate from serving user queries. Getting it wrong here cannot be fixed downstream — garbage in, garbage out is nowhere more true than in RAG indexing.

1

Document ingestion — PDFs, Word, HTML, Markdown all arrive as unstructured bytes

Naive PDF text extraction corrupts tables. Use a layout-aware parser (e.g. AWS Textract). Scanned documents need OCR first.

2

Metadata extraction — title, author, department, last-modified date, access tier

This becomes a filtering layer at query time. You cannot add it later without re-processing the corpus.

3

Chunking — split documents into retrievable pieces at three granularities

Sentence (~100 tokens) for precise lookups · Paragraph (~400 tokens) for most questions · Section (~900 tokens) for context expansion

4

Embedding — each chunk → dense vector via a text embedding model

OpenAI text-embedding-3-large is a strong default. Domain-specific corpora (legal, medical, engineering) may need fine-tuning.

5

Indexing — write vectors into HNSW (dense) + BM25 (sparse) indexes

Two indexes, two retrieval strategies. Dense catches semantic meaning; sparse catches exact product codes and error IDs.

Common beginner mistake: using fixed-size chunking without overlap. A sentence that answers the question may sit exactly on a chunk boundary and get split in half, making it un-retrievable.

Module 3 — The online pipeline

What happens every time a user asks a question

This pipeline must complete in under 5 seconds end-to-end. Every stage adds latency, so every stage must earn its place. The order is non-negotiable — you cannot re-rank before you retrieve.

1

Query understanding — rewrite the user's colloquial question into search-friendly form

Technique: HyDE (Hypothetical Document Embedding) — generate a short hypothetical answer and embed that instead of the raw question.

2

Intent classification — is this query in scope at all?

A lightweight classifier catches off-topic questions before any expensive retrieval runs.

3

Access control filter — scope retrieval to the user's permission set

Enforced server-side before any index query runs. Chunk-level tags, not document-level.

4

Hybrid retrieval — run dense ANN + sparse BM25 in parallel, fuse with RRF

Reciprocal Rank Fusion combines rankings by position, not by raw score. Result: ~80 fused candidates.

5

Cross-encoder re-ranking — score each candidate jointly with the query

A cross-encoder reads query and document together, allowing token-level attention. MiniLM scores 80 candidates in ~300ms. Output: top 5–10.

6

Context assembly — expand + compress the retrieved chunks

Expansion: pull neighboring chunks. Compression: extractive model scores each sentence by relevance. Less context = fewer hallucinations.

7

LLM generation — stream the answer with mandatory citations

System prompt: "answer only from provided context; if the answer isn't there, say so."

Mental model: think of retrieval as casting a wide net (top 80 candidates) and re-ranking as a careful hand-sort of the catch. The LLM is the chef who turns the best fish into a meal — but only what's on the plate in front of it.

Module 4 — Guardrails

Why production RAG systems fail silently — and how to stop it

Most RAG hallucinations happen not because the LLM makes things up unprompted, but because retrieval failed and the model was handed a loosely relevant context and told to fill in the gaps.

Key insight: faithfulness scoring runs in parallel with response streaming — the NLI model checks each completed sentence while the LLM is generating the next one. The user perceives zero added latency.

Contradiction handling: if two retrieved chunks conflict (e.g. one policy says 15 days, a newer one says 12), the LLM will try to silently average them. The assembly layer must detect same-title/different-timestamp documents and explicitly surface the conflict.

Module 6 — Online monitoring & continuous improvement

A RAG system at launch is not a finished product

The query distribution shifts as the company changes. Documents go stale the moment they are updated. The system that ships is the worst version it will ever be — if you close the loop correctly.

1

Query drift monitoring — track embedding distribution of incoming queries with a rolling 30-day window

A drift spike means users are asking about topics not yet covered in the corpus.

2

Per-query quality metrics — rolling faithfulness rate, no-answer rate, P95 latency by pipeline stage

P95 latency must be broken down by stage. If P95 degrades, you need to know which stage is the bottleneck.

3

Corpus health probes — after every weekly batch, run known QA pairs against the top 100 most-queried documents

If Recall@K drops on a high-traffic document after an update, the update silently broke retrieval for that topic.

4

Fine-tuning feedback loop — use labeled production signals to periodically retrain the embedding model and re-ranker

Exposure bias caution: counter with hard negative mining and epsilon-greedy exploration.

The loop in one sentence: monitor for drift → identify the failing stage → trace it to corpus gap or technical regression → fix the right thing → re-evaluate → ship.
