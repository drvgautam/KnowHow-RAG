The Offline Indexing Pipeline

How raw documents become searchable chunks — the batch process that runs before any query is served

Why a separate offline pipeline?

The offline pipeline processes your document corpus into a searchable index. It runs on a schedule (daily, weekly) and is completely decoupled from user queries. This separation means you can re-index without downtime and optimize each pipeline independently.

Getting this pipeline wrong cannot be fixed downstream. If a document is poorly chunked or its metadata is missing, no amount of re-ranking or prompt engineering will recover the lost information.

The five stages

  1. Document ingestion: PDFs, Word, HTML, Markdown arrive as unstructured bytes. Naive PDF extraction corrupts tables — use layout-aware parsers (AWS Textract, Unstructured.io). Scanned documents need OCR first.
  2. Metadata extraction: Title, author, department, last-modified date, access tier. This becomes a filtering layer at query time. You cannot add metadata later without re-processing the entire corpus.
  3. Chunking: Split documents into retrievable pieces. Sentence-level (~100 tokens) for precise lookups, paragraph-level (~400 tokens) for most questions, section-level (~900 tokens) for context expansion. Always use 10-20% overlap.
  4. Embedding: Each chunk is converted to a dense vector using an embedding model. OpenAI text-embedding-3-large or Cohere embed-v3 are strong defaults. Domain-specific corpora may need fine-tuned embeddings.
  5. Indexing: Write vectors into both HNSW (dense) and BM25 (sparse) indexes. Two indexes enable hybrid retrieval — dense catches semantic meaning, sparse catches exact identifiers and product codes.

Document format challenges

  • Tables: Require special handling — each chunk must include column headers. A table row without its header is meaningless.
  • Images with text: Diagrams, flowcharts, and screenshots need OCR or vision models to extract embedded text.
  • Nested documents: A PDF with embedded spreadsheets or a Word doc with linked files must be recursively unpacked.
  • Encoding issues: Legacy documents in non-UTF8 encodings corrupt silently. Always detect and convert encoding before processing.

Incremental vs. full re-indexing

Full re-index: Process the entire corpus from scratch. Simple, but expensive for large corpora (millions of documents).

Incremental: Only process new or modified documents. Track document hashes to detect changes. Requires careful handling of deletions — removing a document means removing all its chunks from the index.

Never skip metadata extraction to save time. Without department tags and access tiers, you cannot implement access control at query time — and adding it later requires a full re-index of every document.
