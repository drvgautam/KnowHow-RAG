Bi-encoder vs Cross-encoder

The fundamental speed-accuracy trade-off in information retrieval

Bi-encoder (for retrieval)

A bi-encoder processes the query and each document independently. It produces a vector for the query and a vector for each document, then computes similarity (e.g., cosine distance) between them.

  • Document vectors can be pre-computed and cached
  • At query time, only the query needs to be encoded
  • Similarity lookup via ANN index takes milliseconds
  • Scales to millions of documents

Weakness: Because query and document are encoded separately, the model cannot attend to interactions between specific query words and specific document words.

Cross-encoder (for re-ranking)

A cross-encoder processes the query and document together as a single input. It sees the full interaction between query tokens and document tokens through self-attention.

  • Much more accurate relevance scoring
  • Can catch nuances like negation, conditional statements, specific entity matches
  • Cannot pre-compute — must run for every (query, document) pair at query time
  • Too slow for initial retrieval, but fast enough for re-ranking 50-100 candidates

The two-stage pattern

Production RAG systems use both:

  • Stage 1 — Retrieval (bi-encoder): Fast, approximate. Returns top ~80 candidates from millions.
  • Stage 2 — Re-ranking (cross-encoder): Slow, precise. Scores the 80 candidates and keeps the top 5-10.

This gives you the best of both worlds: the speed of bi-encoders with the accuracy of cross-encoders.

Latency comparison

Bi-encoder retrieval:  ~10ms (ANN lookup)
Cross-encoder re-rank: ~300ms (80 candidates)
Total:                 ~310ms
Cross-encoder on full corpus: 10M docs x 3ms each
                              = ~8.3 hours per query
                              (not viable)

The MiniLM cross-encoder (22M parameters) achieves near-state-of-the-art re-ranking accuracy while being small enough to run on CPU. It's the most common choice for production RAG re-ranking.
