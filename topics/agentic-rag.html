Agentic RAG

When the system reasons about what to retrieve, not just how

Beyond single-shot retrieval

Standard RAG follows a fixed pipeline: embed query, retrieve, re-rank, generate. Agentic RAG adds a reasoning loop — the LLM decides what to search for, evaluates the results, and may search again with a refined query before generating the final answer.

The agent loop

  1. Plan: The agent analyzes the user's question and determines what information it needs. A complex question like "Compare our parental leave with industry standards" might require two separate retrievals.
  2. Retrieve: The agent issues one or more targeted searches, potentially against different knowledge bases or with different query formulations.
  3. Evaluate: The agent assesses whether the retrieved context is sufficient. If not, it reformulates and retrieves again.
  4. Synthesize: Once satisfied with the evidence, the agent generates the final answer with citations.

Key patterns

  • Multi-step retrieval: Break complex questions into sub-queries. "How did our revenue change after launching feature X?" requires retrieving both revenue data and the feature launch timeline.
  • Self-RAG: The model generates retrieval tokens that trigger searches mid-generation. It decides on-the-fly when it needs more evidence, rather than front-loading all retrieval.
  • Tool-augmented RAG: The agent can call APIs, run SQL queries, or execute code in addition to searching documents. The knowledge base is just one tool among many.
  • Corrective RAG (CRAG): After initial retrieval, a lightweight evaluator scores relevance. Low scores trigger query refinement and re-retrieval before generation.

When to use agentic RAG

  • Good fit: Multi-hop questions, comparative analysis, questions requiring multiple data sources, research-style queries.
  • Poor fit: Simple factoid lookups ("What's the phone number for HR?"), latency-critical applications, high-volume support chatbots.

Cost and latency trade-offs

Each agent loop iteration adds 1-3 seconds and one LLM call. A 3-step agent loop costs 3x the tokens and 3x the latency of single-shot RAG. Use it selectively — route simple questions to the fast path and complex questions to the agent path.

Agent loops can get stuck in infinite retrieval cycles. Always set a maximum iteration count (typically 3-5) and a total latency budget. If the agent hasn't found sufficient evidence by the limit, it should say so rather than keep searching.
