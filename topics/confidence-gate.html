Retrieval Confidence Gate

The most important guardrail — stop before calling the LLM

The core idea

After retrieval and re-ranking, check whether the top results are actually relevant. If the best retrieved chunk scores below a confidence threshold, do not send it to the LLM. Instead, return a controlled "I don't have information on that" response.

Why this is the most important guardrail

Most hallucinations in RAG systems don't come from the LLM making things up unprompted. They come from the LLM being given loosely relevant context and doing its best to answer from it. A passage about "employee benefits" might be retrieved for a question about "parental leave" — and the LLM will extrapolate an answer that sounds reasonable but isn't in the source.

The confidence gate prevents this entire class of failures.

How to implement it

  • Use the cross-encoder score from re-ranking as the confidence signal — it's already computed, and it measures query-document relevance directly.
  • Set a threshold based on your evaluation set. Typical starting point: if the top result's cross-encoder score is below 0.5, gate it.
  • Calibrate with production data: Run the gate in shadow mode first (log decisions but don't enforce), review the gated queries, and adjust the threshold.

What to say when the gate triggers

Never say "I don't know" — that's ambiguous. Instead, be specific:

  • "I couldn't find relevant information in the knowledge base for this question."
  • "This topic doesn't appear to be covered in the available documentation. You may want to contact [department]."
  • Optionally: show the user what was found, with a warning that it may not be relevant.

A "sorry, I can't answer that" response costs 10ms. A hallucinated answer costs user trust. The confidence gate is a cheap insurance policy against the most expensive failure mode.
