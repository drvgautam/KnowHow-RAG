RAG Monitoring & MLOps

A RAG system at launch is the worst version it will ever be — if you close the loop correctly

Why monitoring matters

The query distribution shifts as the company changes. Documents go stale the moment they're updated. User expectations evolve. Without monitoring, you won't know your system is degrading until users start complaining — and by then, trust is already eroded.

Four pillars of RAG monitoring

  1. Query drift detection: Track the embedding distribution of incoming queries with a rolling 30-day window. A drift spike means users are asking about topics not yet covered. Implement using cosine distance between the centroid of this week's queries and last month's.
  2. Per-stage latency tracking: Break P95 latency down by pipeline stage (retrieval, re-ranking, generation). If P95 degrades, you need to know which stage is the bottleneck immediately — not after a week of investigation.
  3. Quality metrics: Track rolling faithfulness rate, no-answer rate, and citation accuracy. A spike in no-answer rate could be a corpus gap (users asking about something undocumented) or a retrieval regression (indexing broke). You tell them apart by checking Recall@K on your eval set.
  4. Corpus health probes: After every weekly batch, run known QA pairs against the top 100 most-queried documents. If Recall@K drops on a high-traffic document after an update, the update silently broke retrieval for that topic. Probes catch it within hours rather than weeks.

The feedback loop

  • Implicit signals: User reformulations (the first answer was wrong), thumbs up/down, click-through on citations, session abandonment.
  • Explicit signals: "Was this helpful?" ratings, error reports, user corrections.
  • Closing the loop: Aggregate signals into labeled training data. Periodically retrain embedding models and re-rankers on production-derived labels.

Alerting and SLOs

Define service level objectives for your RAG system:

  • P95 end-to-end latency < 5 seconds
  • Faithfulness rate > 0.90 (rolling 7-day)
  • No-answer rate < 15% (excluding truly out-of-scope queries)
  • Retrieval Recall@10 > 0.85 on eval set (checked after each re-index)

Alert when any SLO is breached for more than 2 consecutive measurement windows.

A/B testing RAG changes

Every pipeline change (new chunking strategy, updated embedding model, different re-ranker) should be A/B tested:

  • Run the new pipeline in shadow mode alongside production
  • Compare retrieval metrics on the same queries
  • Only promote to production when the eval set shows improvement and no regressions

The improvement loop in one sentence: monitor for drift, identify the failing stage, trace it to corpus gap or technical regression, fix the right thing, re-evaluate, ship.
