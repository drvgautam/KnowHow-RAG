HNSW Explained

How approximate nearest neighbor search makes RAG fast

The problem

If you have 10 million document chunks, each represented as a 1536-dimensional vector, finding the most similar vector to a query by comparing against all 10 million would take too long for real-time use. You need a way to find the approximate nearest neighbors in milliseconds, not seconds.

What is HNSW?

Hierarchical Navigable Small World (HNSW) is the most widely used algorithm for approximate nearest neighbor (ANN) search. It builds a multi-layer graph where:

  • Bottom layer: Contains all vectors, connected to their nearest neighbors.
  • Upper layers: Contain progressively fewer vectors, acting as "express lanes" for navigation.

How search works

  • Start at the top layer (fewest nodes, longest jumps)
  • Greedily navigate toward the query vector
  • Drop to the next layer and refine
  • Repeat until the bottom layer, where the final nearest neighbors are found

This typically finds the true nearest neighbors 95-99% of the time, while only examining a tiny fraction of the total vectors.

Key parameters

  • M (connections per node): Higher M = more accurate but more memory. Default: 16.
  • ef_construction: How many candidates to consider when building the graph. Higher = better index quality, slower build.
  • ef_search: How many candidates to consider at query time. Higher = more accurate, slower search. This is tunable at runtime.

HNSW in practice

Most vector databases (Pinecone, Weaviate, Qdrant, pgvector) use HNSW as their default index type. Typical performance:

  • 10M vectors: ~5ms per query
  • 100M vectors: ~15ms per query
  • Memory: ~1.5GB per million 1536-dim vectors

HNSW is like looking up a word in a dictionary. You don't read every page — you jump to roughly the right section (top layer), then narrow down by page (middle layers), then scan the entries (bottom layer). The multi-layer structure is what makes it fast.
