Building Evaluation Sets

How to bootstrap a golden eval set without months of manual labeling

What is a golden eval set?

A set of (question, expected answer, source document) triples that you use as ground truth to evaluate your RAG pipeline. You need at least 100-200 examples for statistically meaningful evaluation; 500+ is better.

The LLM-assisted approach

  • Step 1: Select 50-100 representative documents from your corpus (cover different topics, formats, difficulty levels).
  • Step 2: For each document, use an LLM to generate 3-5 questions that the document answers. Prompt: "Given this passage, generate questions that a user might ask that this passage would answer."
  • Step 3: For each question, use the LLM to generate a reference answer with citations to the source passage.
  • Step 4: Have a human review and correct the generated QA pairs. This is 5x faster than writing them from scratch.

Question diversity

Your eval set must cover different question types:

  • Factoid: "What is the deductible for Plan A?" (single fact lookup)
  • Multi-hop: "How does Plan A's deductible compare to Plan B's?" (requires two documents)
  • Conditional: "Am I eligible for parental leave if I'm part-time?" (requires understanding conditions)
  • Unanswerable: "What is the dress code?" (no document covers this — tests the confidence gate)
  • Ambiguous: "What's the policy on leave?" (too vague — tests query clarification)

Maintaining the eval set

The eval set must evolve with your corpus. When documents are updated, the corresponding QA pairs may become stale. Include a "last verified" date on each example and re-verify on a quarterly cycle.

Think of the eval set as your unit tests. You wouldn't ship software without tests, and you shouldn't ship a RAG system without an eval set. Just as tests catch regressions in code, the eval set catches regressions in retrieval and generation quality.
